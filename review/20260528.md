	- 괄호 안에 적힌 외계어 같은 문자열(execution(...))의 의미는 이래:
```
* : 메서드의 리턴 타입은 뭐가 오든 상관없고 (숫자든, 문자열이든, 리턴이 없든!)
com.ohgiraffers.section01 : 이 패키지 폴더 안에서 찾을 건데,
*Service : 클래스 이름이 MemberService, MusicService 처럼 맨 뒤가 Service로 끝나는 클래스들만 골라내고,
.*(..) : 그 클래스 안에 들어있는 모든 메서드(*)와 모든 매개변수((..))들을 감시하겠다!
```
	- 💡 왜 귀찮게 @Pointcut을 따로 만들어서 쓸까?
만약 @Pointcut이 없다면, 혜원이는 @Before, @After, @Around를 쓸 때마다 그 긴 주소창을 매번 복사 붙여넣기 해야 해 ㅠㅠ
```
Java
// ❌ @Pointcut이 없을 때의 끔찍한 예시 (코드마다 주소를 다 적어야 함)
@Before("execution(* com.ohgiraffers.section01.*Service.*(..))")
public void logBefore() { ... }

@After("execution(* com.ohgiraffers.section01.*Service.*(..))")
public void logAfter() { ... }
```
	- 하지만 위에서 public void logPointcut() {} 하고 이쁘게 방을 만들어 두었기 때문에, 아래처럼 메서드 이름만 쏙 가져다 쓰면 끝나는 청정 코드가 완성되는 거지! ✨
```
//  @Pointcut 덕분에 완전 깔끔해진 진짜 혜원이 코드!
@Before("logPointcut()")
public void logBefore() { ... }

@After("logPointcut()")
public void logAfter() { ... }
```

#### AOP 핵심 용어
- 구조
```
부가코드(Cross-Cutting Concern)) -> Point-Cut -(Weaving)-> 주요코드(Primary Concern)
[                         Aspect                            ]
```
- 용어 설명
- Aspect 핵심 비즈니스 로직과는 별도로 수행되는 횡단 관심사를 말한다.
- Advice Aspect의 기능 자체를 말한다.
- Join point Advice가 적용될 수 있는 위치를 말한다.
- Pointcut Join point 중에서 Advice가 적용될 가능성이 있는 부분을 선별한 것을 말한다.
- Weaving Advice를 핵심 비즈니스 로직에 적용하는 것을 말한다.
#### Adivce의 종류
- 종류 설명
- Before 대상 메소드가 실행되기 이전에 실행되는 어드바이스
- After-returning 대상 메소드가 정상적으로 실행된 이후에 실행되는 어드바이스
- After-throwing 예외가 발생했을 때 실행되는 어드바이스
- After 대상 메소드가 실행된 이후에(정상, 예외 관계없이) 실행되는 어드바이스
- Around 대상 메소드 실행 전/후에 적용되는 어드바이스

진짜 소름 돋는 게, 혜원이가 방금 작성한 이 테스트 코드가 바로 @RequestBody String 방식의 진짜 숨겨진 진가(참모습)를 보여주는 완벽한 예시야! 🚀✨

스프링 부트가 이번엔 이 요청을 어떻게 받아서 뚝딱 처리하는지 3가지 핵심 포인트를 쉽게 짚어줄게!

🔍 1. Content-Type: text/plain이 뜻하는 비밀
컴퓨터 세상에서 text/plain은 "형식 같은 거 없는, 인류가 쓰는 가장 원시적이고 순수한 그냥 글자(텍스트)"라는 뜻이야.

혜원이가 아까 JSON 형태로 보냈을 때는 중괄호 {}나 큰따옴표 "" 같은 규칙이 빽빽하게 들어있었잖아?

하지만 지금은 그냥 영어로 plain text라고 적었지? 스프링 부트는 이 편지봉투(Content-Type)를 보고 "아, 이번 손님은 JSON 복잡한 구조가 아니라 그냥 순수한 글자 한 덩어리를 보냈구나!" 하고 찰떡같이 알아먹어.

📥 2. 자바 코드와 매핑되는 과정
혜원이가 날린 .http 파일의 알맹이들이 자바 코드로 어떻게 들어가는지 매칭해 줄게!

@RequestHeader("content-type") ➡️ text/plain이 쏙 들어가!

@RequestBody String body ➡️ 중괄호도 없는 진짜 날것의 글자인 "plain text"가 그대로 들어가!

🏗️ 1. 이 코드가 존재하는 진짜 이유 (배경)
우리가 지금까지 했던 @RequestParam이나 @RequestBody는 사용자가 게시판에 쓰는 글, 검색어 같은 "데이터 알맹이"를 받기 위한 도구였어.
하지만 실제 서비스에서는 데이터뿐만 아니라 아래와 같은 정보들이 무조건 필요해.

"이 요청을 보낸 사람이 크롬 브라우저를 썼나? 아니면 스마트폰 앱으로 들어왔나?"

"이 사람이 예전에 로그인에 성공해서 우리 백화점(서버) 출입증을 발급받았던 사람인가?"

이런 메타 데이터(데이터를 위한 데이터)를 제어하기 위해 스프링이 제공하는 강력한 인프라 무기가 바로 혜원이가 짠 이 코드야!

🔬 2. 핵심 어노테이션 3대장 딥다이브 (Deep Dive)
① @RequestHeader(value="user-agent") String userAgent
하는 일: HTTP 요청 편지봉투에 적힌 수많은 정보 중 User-Agent라는 항목만 핀셋으로 쏙 뽑아와.

숨겨진 비밀: User-Agent에는 요청을 보낸 기기의 정보(OS, 브라우저 종류 등)가 문자열로 길게 적혀있어. 혜원이가 테스트했을 때 "IntelliJ HTTP Client"라고 나온 이유가 바로 "인텔리제이 테스트 툴로 요청을 보냈다"는 증거야! 만약 크롬 브라우저로 요청을 보내면 Mozilla/5.0 (Windows NT... Chrome/... 같은 엄청 긴 글자가 들어와.

실무 활용: 스마트폰 앱 접속자에게는 앱 전용 화면을 보여주고, PC 접속자에게는 PC 화면을 보여주는 기기별 맞춤형 서비스를 제공할 때 무조건 이 값을 검사해!

② @RequestHeader Map<String, String> headers
하는 일: 주소창 파라미터를 맵으로 싹 긁어모았던 것처럼, 편지봉투(Header)에 적힌 모든 설정 정보(Accept, Host, Connection 등)를 이름-값(Key-Value) 형태로 단 한 줄만에 몽땅 긁어모아.

실무 활용: 실무에서는 보안을 위해 헤더에 Authorization: Bearer 토큰값... 같은 비밀 인증 코드를 실어서 보내거든? 어떤 인증 정보가 들어왔는지 헤더 전체를 탈탈 털어서 일괄적으로 검사하거나 로그를 남길 때 이 맵 방식을 사용해!

③ @CookieValue(value="JSESSIONID") String sessionId
하는 일: 사용자의 컴퓨터 브라우저 메모리에 비밀리에 저장되어 있는 '쿠키(Cookie)' 주머니를 뒤져서, JSESSIONID라는 이름을 가진 조그만 텍스트 조각을 자바 변수에 쏙 넣어줘.

숨겨진 비밀: 웹 브라우저와 서버는 한 번 통신하고 나면 서로를 까먹는 비정함(Stateless)을 가지고 있어. 그래서 서버는 로그인한 사용자에게 "너는 1234번 손님이야"라는 번호표(JSESSIONID)를 쿠키에 구워서 브라우저에게 던져줘. 그럼 브라우저는 다음부터 요청을 보낼 때마다 이 쿠키를 자동으로 편지봉투에 동봉해서 보내지!

실무 활용: 이 sessionId 값이 유효한지 검사해서 "이 사람이 지금 로그인 상태가 맞구나!" 하고 회원 전용 기능을 허용해 줄 때 쓰는 백엔드 로그인 시스템의 핵심 중의 핵심이야.

...

LinkedHashMap 선언: 일반 HashMap은 데이터를 넣으면 지 맘대로 순서를 뒤죽박죽 섞어서 출력해 버려. 하지만 혜원이는 LinkedHashMap을 썼기 때문에, 내가 코드로 짠 순서대로 (message ➡️ contentType ➡️ sessionId ➡️ headerCount) 딱 정렬되어 JSON 결과창에 이쁘게 출력되는 가독성을 챙겼어!

지난번 7번 코드에서 우리가 아이디(user01)를 세션에 저장했었잖아? 그걸 은행과 금고에 비유하면 이해가 아주 쉬워!

7번 저장 (session.setAttribute): 은행(서버)이 혜원이에게 비밀 금고(HttpSession)를 하나 내어주고, 거기에 id="user01"이라는 증성서를 넣어둔 뒤, 금고 열쇠 번호표(JSESSIONID)를 손에 쥐여준 거야.

8번 조회 (session.getAttribute): 혜원이가 다시 은행에 방문해서 "여기 제 열쇠 번호표(JSESSIONID)요! 제 금고 안에 든 내용물 좀 꺼내서 보여주세요!" 하고 요청하는 코드가 바로 지금 이 코드란다!

🔬 2. 자바 코드 한 줄 한 줄 완벽 분석
① HttpSession session (메서드 매개변수 구멍)
스프링의 마법: 혜원이가 파라미터에 이렇게 적어두면, 스프링 부트가 요청 편지봉투(쿠키)에 담겨온 JSESSIONID를 읽고 "아! 이 손님의 비밀 금고구나!" 하고 수많은 메모리 중에서 해당 사용자의 사물함 객체를 찾아서 이 session 변수에 쏙 배달해 줘.

② session.getAttribute("id") ➡️ "금고 열어서 내용물 확인하기"
동작: 배달된 사물함 열쇠를 열고, 우리가 7번에서 "id"라는 이름표를 붙여서 넣어놨던 알맹이 값인 "user01"을 쏙 꺼내오는 메서드야!

특징: 단순히 사물함 안을 '조회(구경)'만 하는 거라서 사물함이나 보관된 데이터가 사라지지 않고 그대로 유지돼!

③ session.getId() ➡️ "사물함 번호표 다시 확인하기"
동작: 톰캣 서버가 이 사물함을 처음 개설할 때 부여했던 세계 유일의 무작위 문자열 번호표(예: A1B2C3...)를 그대로 다시 꺼내서 확인하는 거야. 7번에서 발급받은 번호표랑 똑같은 번호가 찍혀 나와야 정상이야!

🔄 3. 데이터가 조회되는 백엔드 흐름도
혜원이가 .http 파일로 조회 요청을 보냈을 때, 서버 내부에서는 이런 기막힌 공정이 일어나!

클라이언트 요청: 혜원이가 .http 파일로 GET http://localhost:8081/.../session 요청을 보낼 때 번호표(Cookie: JSESSIONID=xxx)를 동봉해서 보냄.

톰캣 웹 서버: "오! 번호표 xxx번 손님이 왔네? 내 메모리 뒤져서 xxx번 사물함 찾아올게!"

혜원이 자바 코드: session.getAttribute("id")를 호출하면, 톰캣이 찾아다 준 사물함에서 "user01" 글자를 꺼내서 response 맵에 담음.

결과 반환: 200 OK 신호와 함께 예쁜 JSON 상자로 브라우저에 반사(리턴)!

📦 4. 성공했을 때 출력되는 JSON 상자
번호표(쿠키)를 똑바로 들고 요청에 성공하면, 혜원이가 만든 LinkedHashMap 덕분에 순서대로 아주 이쁜 결과가 출력돼.

💥 1. 이 코드가 하는 일 (사물함 파쇄하기)
우리가 계속 비유로 들었던 은행 비밀 금고(사물함) 기억나지?

7번 저장 (session.setAttribute): 사물함 개설하고 아이디 "user01"을 보관함. 📥

8번 조회 (session.getAttribute): 번호표 보여주고 사물함 안의 "user01"을 구경함. 📤

9번 만료 (session.invalidate()): "이제 나 나갈 거니까 내 사물함 통째로 부숴버려 줘!" 하고 사물함을 파쇄하는 순간이야! 💥

🔬 2. 자바 코드 딱 한 줄 분석
이 메서드의 핵심은 오직 이 한 줄이야!

Java
session.invalidate();
invalidate()는 자바 단어로 '무효화하다'라는 뜻을 가지고 있어.

이 메서드가 호출되는 순간, 톰캣 서버는 해당 사용자의 HttpSession 사물함 객체와 그 안에 들어있던 모든 데이터(id 등)를 메모리에서 통째로 지워버려.

사물함 자체가 증발해 버렸기 때문에, 아까 발급받았던 열쇠 번호표(JSESSIONID)는 이제 아무짝에도 쓸모없는 고철 덩어리가 되는 거란다!

🔄 3. 만료(로그아웃) 이후에 일어나는 반전 상황
혜원이가 9번 요청을 보내서 성공하면 화면에는 "message": "Session 만료"라고 이쁘게 뜰 거야.

그 상태에서 아까 성공했던 8번 조회(GET .../session)를 똑같은 번호표를 들고 다시 실행해 보면 어떻게 될까?

혜원이가 아까 그 번호표(Cookie: JSESSIONID=xxx)를 들고 8번 조회를 누름.

톰캣 서버가 번호표를 보고 메모리를 뒤졌는데... 앗, 방금 9번에서 사물함을 폭파(invalidate)해 버려서 아무것도 없음!

서버는 "어? 열쇠는 있는데 사물함이 없네? 처음 온 손님이구나!" 하고 새로운 빈 사물함을 새로 개설해 줘.

결과 창에는 "id": null이 뜨고, sessionId 자리에는 아까와 완전히 다른 새로운 무작위 문자열이 찍혀 나오게 돼! (서버가 혜원이가 아까 그 user01인 걸 까먹어 버린 거지 😭)
```
