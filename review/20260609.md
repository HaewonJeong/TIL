#### MapperScan 어노테이션

```
@MapperScan(
     basePackages = "com.ohgiraffers.springmybatis", 
     annotationClass = Mapper.class
) 
```
의 의미는 com.ohgiraffers.springmybatis` 패키지 아래에서  > `@Mapper`가 붙은 인터페이스를 찾아서 MyBatis Mapper Bean으로 등록 해라 의미

- return ResponseEntity.status(HttpStatus.CREATED).body(registMenu);
	- CREATED 201 응답 & 등록된 메뉴를 응답 body에 넣어서 반환

- JSON → DTO Setter → Mapper 전달 → XML의 #{필드명} → DTO Getter → SQL 실행

- Transitional 어노테이션 : 예외상황 없이 호출이 되면 자동 commit.&문제 상황 시 자동  rollback
	- INSERT, UPDATE, DELETE 같은 DML 작업이 성공하면 COMMIT, 실패하면 ROLLBACK 되도록 트랜잭션을 관리하는 어노테이션
	- Transactional 없으면 중간에 예외가 발생했을 때 일부만 반영될 수 있음
- Mybatis 실습 코드 주석
```
<insert id="registMenu" parameterType="com.ohgiraffers.springmybatis.menu.model.dto.MenuDTO"  
    useGeneratedKeys = "true" keyProperty="code" >  
    insert into tbl_menu  
    (menu_name, menu_price, category_code, orderable_status)  
    values  
    ( #{ name }, #{ price }, #{ categoryCode }, #{ orderableStatus } )  
</insert>
```
- <MenuDTO의 getPrice~ getName~ getter에서 값을 받아온다.
- keyProperty="code": DB가 자동 생성한 key값을 다시 받아온다.
- keyProperty="code" 받아온 key 값을 MenuDTO의 code 프로퍼티에 넣는다.
- useGeneratedKeys="true" : INSERT 후 DB가 생성한 AUTO_INCREMENT 값을 가져와라
- keyProperty="code" : 받아온 AUTO_INCREMENT 값을 DTO의 code 필드에 넣어라

- JSON에 요청하지 않는 int 변수를 Integer로 선언 이유
	- private int code; 일때, JSON에 code가 없으면 Jackson이나 MyBatis가 객체를 만들면서 code=null 상태를 처리해야 하는 순간이 있다. int는 원시형이라 null을 담을 수 없다. 반면 private Integer code;는 Integer code = null;이 가능하다.
- int deleteMenu(@Param("menuCode") int menuCode);
	- Mapper XML에서 무조건 `menuCode`라는 이름으로 사용할 수 있도록 매개변수에 이름을 붙여준다.
- soft delete 
	- 데이터를 직접 DELETE 하지 않고 상태값을 관리하는 방식
		- selete 시 state = 'Y' 만 조회
		- deletemapping 시 update state = 'N' 변경
- 어떤 상태 코드를 보낼지는 컨트롤러가 결정하고 상태코드도 잘 반환해줘야 한다.

##### JPA 사용 이유
1. 데이터 변환, SQL 작성, JDBC API 코드 등을 중복 작성
2. SQL 의존적인 개발
	1. DB에서 모든 기능을 수행하고 결과를 옮기기만 함
3. 패러다임 불일치
	1. 객체 지향 개념을 활용하지 못함
- 4. 동일성 보장 문제
	- 같은 쿼리를 두번 조회 시 주소가 다르다.

##### ORM (Object Relational Mapping) 개요

1. 객체(Java)와 관계형 DB(Table)를 자동으로 매핑해주는 기술
2. SQL을 직접 많이 작성하지 않고 객체 중심으로 개발 가능
3. 예: `MenuDTO` ↔ `tbl_menu`

```
Java 객체 ↔ ORM ↔ DB 테이블
```

---

##### JPA (Java Persistence API) 개요

1. Java에서 ORM을 사용하기 위한 표준 명세(인터페이스)
2. 객체를 저장, 조회, 수정, 삭제하는 방법을 정의
3. 대표 구현체: Hibernate
```
JPA = ORM을 위한 Java 표준 Hibernate = JPA 구현체
```

#### EntityManager

1. JPA에서 **엔티티(Entity)를 관리하는 핵심 객체**
2. DB에서 조회한 데이터를 **Entity 객체로 관리하고 추적**한다.
3. 저장, 조회, 수정, 삭제(CRUD)를 수행하며 트랜잭션과 영속성 컨텍스트를 관리한다.
##### EntityManager Factory
- 엔터티 매니저를 생성할 수 있는 기능을 제공하는 팩토리 클래스
- 프로그램 시작할 때 1개만 만들고, EntityManager, PersistenceContext가 요청이 있을때 마다 쌍으로 생성된다.
##### PersistenceContext
- EntityManager가 생성되면 PersistanceContext도 하나씩 생성되고, 이 저장 공간을 다루는 것은 EntityManager가 한다.
- Key - Value의 형태로 관리 되서 저장된다.
