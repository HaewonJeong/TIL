- @Results
- DB 컬럼과 DTO 필드를 연결해주는 매핑 규칙.
```
<resultMap id="menuResultMap">
    <id property="code" column="menu_code"/>
    <result property="name" column="menu_name"/>
    <result property="price" column="menu_price"/>
</resultMap>

↓

어노테이션

@Results(id = "menuResultMap", value = {
    @Result(property = "code", column = "menu_code", id = true),
    @Result(property = "name", column = "menu_name"),
    @Result(property = "price", column = "menu_price")
})

같은 의미.
```


### Section03 
- XML + DAO & 어노테이션 + Mapper
   - Mapper(타입 안정성) + XML(SQL 분리 + 동적 쿼리 가독성)

- Service -> MenuDAO -> SqlSession 문자열 호출
- Service -> MenuMapper 인터페이스 -> XML SQL 실행

- 구조
```
Controller
    ↓
Service
    ↓
DAO
    ↓
Mapper Interface
    ↓
XML(SQL)
    ↓
DB
```
- Mapper 사용 이유 (타입 안정성)
```
#### 기존 방식
sqlSession.selectList("MenuMapper.selectAllMenu");
문자열로 SQL을 호출한다.
문제점:
오타가 있어도 컴파일 시점에는 발견되지 않고 실행 시 오류가 발생한다.
```
#### Mapper 방식
```
MenuMapper mapper = sqlSession.getMapper(MenuMapper.class);
mapper.selectAllMenu();
```
- 인터페이스의 메서드를 호출하므로
 - 컴파일 시점에 오류 확인 가능
 - 오타 방지
 - IDE 자동완성 지원
 - 리팩토링이 쉬움
 -  이를 타입 안정성(Type Safety) 이라고 한다.

####  XML 사용 이유 (동적 SQL)
- 조건에 따라 SQL이 달라질 경우 XML이 훨씬 편리하다.
- 검색 조건에 따라 SQL을 동적으로 생성할 수 있다.
```
<select id="searchMenu">
    SELECT *
    FROM tbl_menu
    <where>
        <if test="name != null">
            AND menu_name LIKE CONCAT('%', #{name}, '%')
        </if>

        <if test="categoryCode != null">
            AND category_code = #{categoryCode}
        </if>
    </where>
</select>
```
##### 어노테이션 방식과 비교
어노테이션 방식
```
@Select("""
    SELECT *
    FROM tbl_menu
    WHERE menu_code = #{code}
""")
MenuDTO selectMenuByCode(int code);
```
- 장점
  - 간단한 SQL 작성이 편함
  - XML 파일이 필요 없음
- 단점
  - SQL이 길어지면 가독성 저하
  - 동적 SQL 작성이 복잡함
- 실무에서는 보통 Mapper 인터페이스 + XML SQL 조합을 가장 많이 사용한다.

##### 질문 :  #{name},는 어디서 가져오나요?
  - Mapper 메서드에 전달한 객체나 Map 안의 name 값을 가져오는 것
  - 예를 들어 DTO를 전달한다고 해보자.
```
SearchCriteria criteria = new SearchCriteria();
criteria.setName("김치");

mapper.searchMenu(criteria);
```
  - Mapper
```
List<MenuDTO> searchMenu(SearchCriteria criteria);
```
  - XML
```
<if test="name != null">
    AND menu_name LIKE CONCAT('%', #{name}, '%')
</if>
```
- 여기서 #{name} 값은 criteria.getName() 값을 가져옴.


### Chapter04
#### MyBatis의 동적 SQL(Dynamic SQL) 에서 사용하는 <if> 태그
- MyBatis XML에서는 >=, <=, >, <를 직접 쓰면 XML과 충돌할 수 있어서 문자로 표현 한다.
```
<if test="price gte 0 and price lte 10000">
</if>
```
- 위 코드의 의미는
```
if(price >= 0 && price <= 10000)

```
- 와 거의 똑같다.

| MyBatis            | 의미          |
| ---------------- | --------------- |
| gt        | >            |
| gte        | >=           |
| lt       | <           |
| lte    | <=    |

#### CDATA
```
<![CDATA[
    SQL
]]>
```
- XML은 <. > &를 그냥 쓰면 태그 시작으로 오해 할 수 있다.
- CDATA 안에 넣으면 여기는 XML로 해석하지 말고 그대로 문자열로 취급해라. 라는 뜻이 된다.

#### typeAliases
- typeAliases는 클래스의 긴 풀패키지명을 짧은 별명(alias)으로 등록하여 XML에서 parameterType, resultType 등을 간결하게 작성할 수 있도록 하는 기능이다.
- 원래 XML에서 DTO를 사용하려면 풀패키지명을 전부 써야해서 길고 오타나기 쉽다. 
```
<select id="selectMenu"
        resultType="com.ohgiraffers.section01.MenuDTO">
```

- 다음과 같이 등록시
```
<typeAliases>
    <typeAlias
        type="com.ohgiraffers.section01.MenuDTO"
        alias="MenuDTO"/>
</typeAliases>
```
- XML에서 다음과 같이 짧게 사용 가능하다.

```<select id="selectMenu"
        resultType="MenuDTO">
```

#### java.io.Serializable

- 객체를 직렬화(Serialization)할 수 있다는 표시(Marker Interface)
- 컴퓨터는 객체를 주소 + 메모리 구조로 관리하기 떄문에, 객체를 바이트 형태로 변환해서 파일 저장, 네트워크 전송, 캐시 저장 할 수 있게 만드는 것을 직렬화라고 한다. 
- 예시

```
import java.io.Serializable;

public class MenuDTO implements Serializable {

    private int code;
    private String name;
}
```
- 이제 MenuDTO 객체를 파일, 네트워크, 세션 등에 저장 가능하다.
- 메서드가 없는 인터페이스(Maker Interface)다.
- 실무에서는 HttpSession 저장, Redis 캐시, 파일 저장 등에 사용한다.
