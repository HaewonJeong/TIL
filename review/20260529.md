```
stream() 이란?
1. memberList.stream()
Collection 인터페이스에 정의된 default 메서드 stream()이 호출되면서, memberList를 기반으로 하는 Stream<MemberDTO> 객체가 생성돼.

2. .filter(member -> member.getNo() == memberNo)
filter 메서드는 자바의 표준 함수형 인터페이스인 Predicate<T>를 매개변수로 받아.

혜원이가 쓴 람다식(->)은 내부적으로 Predicate 인터페이스의 boolean test(T t) 메서드를 익명 구현 객체 형태로 오버라이딩한 구문이야.

조건이 true인 요소만 남긴 새로운 Stream<MemberDTO>를 반환해.

3. .findFirst()
스트림 파이프라인을 닫는 최종 연산이야. 이 시점에 내부 루프가 돌면서 앞의 filter 조건에 맞는 첫 번째 원소를 찾아 나서지.
이때 반환 타입은 MemberDTO가 아니라 Optional<MemberDTO>라는 특수한 래퍼 객체 타입이 돼. (값이 비어있을(null) 가능성을 내포하는 안전 상자야.)

4. .orElseThrow(...)
Optional<T> 객체의 내장 메서드야. 상자 안이 Optional.empty()(비어있음) 상태라면 괄호 안의 Supplier<X> 함수형 인터페이스가 제공하는 예외 객체를 throw하고, 값이 존재하면 알맹이(MemberDTO)를 꺼내서 좌변의 변수에 할당해 주는 역할을 해.
```

```
package com.ohgiraffers.exceptionhandler;


import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.PathVariable;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RestController;

import java.util.ArrayList;
import java.util.List;

@RestController
@RequestMapping("/api/v1/members")
public class MemberController {

    private final List<MemberDTO> members = new ArrayList<>();

    public MemberController() {
        members.add(new MemberDTO(1, "판다", 5));
        members.add(new MemberDTO(2, "다람쥐", 3));
        members.add(new MemberDTO(3, "양", 6));
    }

    @GetMapping
    public ResponseEntity<List<MemberDTO>> findMembers(){
        return ResponseEntity.ok(members);
    }

    @GetMapping("/{memberNo}")
    public ResponseEntity<MemberDTO> findMember(@PathVariable int memberNo){
        MemberDTO foundMember = members.stream()
                .filter(member -> member.getNo() == memberNo)
                .findFirst()
                .orElseThrow(() -> new MemberNotFoundException(memberNo + "번 회원을 찾을 수 없습니다."));

        return ResponseEntity.ok(foundMember);
    }
}
```

```
3. 내부 동작의 핵심: Stream 파이프라인과 예외 던지기 (Exception Throwing)
Java
@GetMapping("/{memberNo}")
public ResponseEntity<MemberDTO> findMember(@PathVariable int memberNo){
    MemberDTO foundMember = members.stream()
            .filter(member -> member.getNo() == memberNo)
            .findFirst()
            .orElseThrow(() -> new MemberNotFoundException(memberNo + "번 회원을 찾을 수 없습니다."));

    return ResponseEntity.ok(foundMember);
}
이 메서드는 주소창 경로에 실려 온 변수(@PathVariable int memberNo)를 가지고 힙 메모리의 리렉션을 뒤지는 저번 시간에 배운 선언형 파이프라인 연산의 연속이야.

members.stream(): 원본 컬렉션인 ArrayList로부터 요소들의 읽기 전용 시퀀스인 Stream<MemberDTO>를 열어젖혀.

.filter(...): Predicate<MemberDTO> 함수형 인터페이스의 람다 조건식인 member.getNo() == memberNo를 만족하는 요소만 필터링하는 중간 연산을 수행해.

.findFirst(): 필터를 거친 요소 중 첫 번째 원소를 낚아채며 파이프라인을 닫는 최종 연산을 수행하지. 이때 데이터가 있을 수도, 없을 수도 있으므로 자바는 안전하게 Optional<MemberDTO> 객체로 결과물을 감싸서 반환해.

.orElseThrow(...) (비즈니스 예외 전략):

성공 시: 만약 memberNo가 2라면 Optional 내부에서 '다람쥐' DTO 객체를 안전하게 꺼내어(unwrap) foundMember 변수에 할당하고, HTTP 200 OK 상태 코드와 함께 직렬화하여 반환해.

실패 시: 만약 장부에 없는 5번 같은 숫자가 들어오면, Optional 내부가 비어있음을 감지하고 혜원이가 아까 컴파일 에러 고쳐가며 정성껏 만든 MemberNotFoundException 이라는 애플리케이션 정의 예외(Application-defined Exception)를 의도적으로 런타임에 유발(Throw)시켜!
```

```
2. @PathVariable 이란? (URI 경로 변수 추출기)
🔍 기술적 정의
"웹 요청 URL 경로(Path)에 포함된 동적 플레이스홀더(Placeholder, {})의 값을 가로채서 메서드의 매개변수(Parameter)로 바인딩(Binding)해 주는 어노테이션"

RESTful API 디자인에서는 조회하고 싶은 자원의 식별자(ID)를 쿼리 스트링(?memberNo=2)이 아니라, URL 경로 자체의 일부(.../members/2)로 표현하는 것을 강력하게 권장해. 이때 경로에 숨어있는 데이터를 쏙 뽑아내 주는 녀석이 바로 이 녀석이야!
```

```
1. 전반전: .orElseThrow()의 내부 메커니즘
아까 2번 메서드에서 .findFirst() 연산의 결과로 반환되는 타입은 일반 객체가 아니라 Optional<MemberDTO>라는 특수한 래퍼(Wrapper) 상자 객체라고 했지?

Optional<T> 클래스의 내장 메서드인 orElseThrow()의 자바 실제 오픈소스 내부 명세(정의부)를 보면 아래와 같이 구현되어 있어.

Java
// 💡 Java Optional 클래스 내부 실제 구현 모양 (이해를 돕기 위한 축약)
public <X extends Throwable> T orElseThrow(Supplier<? extends X> exceptionSupplier) throws X {
    if (value != null) {
        return value; // 상자 안에 값이 있으면 알맹이(MemberDTO)를 그대로 리턴!
    } else {
        throw exceptionSupplier.get(); // 상자 안이 비었으면(null) 전달받은 예외를 발생시킴!
    }
}
람다식 () -> new ... 의 정체:
orElseThrow()의 매개변수를 보면 Supplier<X>라는 자바 표준 함수형 인터페이스를 받고 있어. Supplier는 "매개변수는 없고, 리턴 값만 있는" 자바 인터페이스야.
즉, 혜원이가 쓴 () -> new MemberNotFoundException(...) 코드는 값이 없을 때 비로소 실행되면서 예외 객체를 생성해서 공급(Supply)해 주는 공장 역할을 하는 람다식인 거란다!

2. 후반전: MemberNotFoundException 커스텀 예외의 정체
혜원이가 만든 이 예외 클래스는 자바의 기본 에러가 아니라 "우리 애플리케이션의 비즈니스 상황을 정확히 대변하는 사용자 정의 예외(Custom Exception)"야.

Java
public class MemberNotFoundException extends RuntimeException { // 1. 언체크 예외 상속

    public MemberNotFoundException(String message) {
        super(message); // 2. 부모 생성자 호출
    }

}
① 왜 RuntimeException을 상속받았을까? (중요!)
자바의 예외는 크게 두 가지로 나뉘어.

Checked Exception (명시적 예외): Exception을 바로 상속받음. 코딩할 때 무조건 try-catch나 throws를 강제로 적어야 해서 코드가 지저분해짐.

Unchecked / Runtime Exception (실행 시 예외): 혜원이가 상속받은 RuntimeException 계열이야. 굳이 메서드 옆에 throws를 안 적어도 스프링이 실행 중에 에러를 알아서 위로 던져(Throw) 주기 때문에 modern 자바나 스프링 프레임워크 실무에서는 99% 이 방식을 채택해!

② super(message); 의 역할
생성자 안에서 부모 클래스의 생성자를 호출하는 super() 문법이야.
RuntimeException 부모 상자 안에는 에러 메시지를 보관하는 메모리 공간과 getMessage()라는 메서드가 이미 다 구현되어 있어. 혜원이가 주입한 "5번 회원을 찾을 수 없습니다."라는 글자를 부모 에러 상자에게 고스란히 토스해서 저장해 두는 역할을 한단다!

3. 두 녀석이 만났을 때 일어나는 전체 시나리오 (Runtime Flow)
만약 5번 회원을 요청해서 장부에 데이터가 없는 상황이 발생하면:

파이프라인 최종 연산 결과, 비어있는 Optional.empty() 상자가 반환됨.

orElseThrow() 메서드가 내부적으로 if (value != null) 조건을 검사했으나 null이므로 else 분기로 진입.

혜원이가 매개변수로 던져준 Supplier 공장(() -> ...)의 버튼을 눌러서 MemberNotFoundException 객체를 힙(Heap) 메모리에 새로 뚝딱 생성(new)함.
```
