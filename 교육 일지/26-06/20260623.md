# Spring Security JWT 인증/인가 실습일지

## 학습 주제
JWT 기반 로그인 / 토큰 재발급(Refresh) / 로그아웃 구현

---

## 1. 전체 흐름 개요

```
로그인 (POST /auth/login)
  → accessToken + refreshToken 발급
  → refreshToken은 HttpOnly 쿠키로 응답
  → accessToken은 응답 body로 전달

이후 API 요청
  → accessToken을 Authorization 헤더에 담아 요청

accessToken 만료 시
  → POST /auth/refresh 호출 (쿠키의 refreshToken 사용)
  → 새 accessToken + refreshToken 재발급

로그아웃 (POST /auth/logout)
  → 서버에 저장된 refreshToken 삭제
  → 쿠키도 즉시 만료 처리
```

---

## 2. 로그인 로직 (AuthService.login)

```java
User user = userRepository.findByUsername(request.getUsername())
        .orElseThrow(() -> new BadCredentialsException("올바르지 않은 아이디 혹은 비밀번호"));

if(!passwordEncoder.matches(request.getPassword(), user.getPassword())) {
    throw new BadCredentialsException("올바르지 않은 아이디 혹은 비밀번호");
}
```

### 배운 점
- 비밀번호는 절대 평문 비교 X → `PasswordEncoder.matches(평문, 암호화된값)`으로 검증
- 아이디가 없을 때와 비밀번호가 틀렸을 때 **같은 예외 메시지**를 사용 → 어느 쪽이 틀렸는지 외부에 노출하지 않기 위한 보안 관행

### 토큰 발급 및 저장

```java
String accessToken = jwtTokenProvider.createToken(user.getUsername(), user.getRole().name());
String refreshToken = jwtTokenProvider.createRefreshToken(user.getUsername(), user.getRole().name());

RefreshToken tokenEntity = RefreshToken.builder()
        .username(user.getUsername())
        .token(refreshToken)
        .expiryDate(new Date(System.currentTimeMillis() + jwtTokenProvider.getRefreshExpiration()))
        .build();
refreshTokenRepository.save(tokenEntity);
```

- **refreshToken은 서버에 저장**해야 함 (탈취/위조 방지 위한 대조용)
- 성능 면에서는 Redis가 추천되지만, 실습에서는 RDBMS로 구현
- 만료일은 `현재시각 + refreshExpiration`으로 계산해서 저장

## 3. HttpOnly 쿠키로 refreshToken 전달

```java
private ResponseCookie createRefreshTokenCookie(String refreshToken) {
    return ResponseCookie.from("refreshToken", refreshToken)
            .httpOnly(true)
            .path("/")
            .maxAge(Duration.ofDays(7))
            .sameSite("Strict")
            .build();
}
```

### 핵심 개념 정리

| 속성 | 역할 |
|---|---|
| `httpOnly(true)` | JavaScript에서 쿠키 접근 차단 → **XSS 공격**으로부터 refreshToken 탈취 방지 |
| `sameSite("Strict")` | 다른 출처(사이트)에서의 요청에는 쿠키를 안 보냄 → **CSRF 공격** 완화 |
| `secure(true)` (주석 처리) | HTTPS 환경에서만 쿠키 전송. 운영 환경에서는 반드시 활성화 필요 |
| `maxAge(Duration.ofDays(7))` | 쿠키 자체의 만료 기간 (refreshToken의 JWT 만료시간과 별개로 쿠키 레벨에서도 관리) |

### 왜 accessToken은 쿠키에 안 담고 refreshToken만?
- accessToken은 매 요청마다 헤더에 실어 보내는 짧은 생명주기 토큰 → body로 클라이언트(React)에 전달, 메모리에서 관리
- refreshToken은 탈취되면 위험이 크기 때문에 JS가 접근 못 하는 HttpOnly 쿠키로 격리

## 4. 토큰 재발급 로직 (AuthService.refreshToken)

```java
jwtTokenProvider.validateToken(providedRefreshToken);
String username = jwtTokenProvider.getUsernameFromJWT(providedRefreshToken);

RefreshToken storedToken = refreshTokenRepository.findById(username)
        .orElseThrow(() -> new BadCredentialsException("해당 유저로 조회되는 리프레시 토큰 없음"));

if(!storedToken.getToken().equals(providedRefreshToken)) {
    throw new BadCredentialsException("리프레시 토큰 일치하지 않음");
}

if(storedToken.getExpiryDate().before(new Date())) {
    throw new BadCredentialsException("리프레시 토큰 유효시간 만료");
}
```

### 3단계 검증 과정
1. **토큰 자체의 서명/형식 검증** (`validateToken`) — 위조 여부 확인
2. **DB에 저장된 토큰과 일치 여부 확인** — 탈취된 토큰이거나 이미 폐기된 토큰인지 체크
3. **DB에 저장된 만료일 재확인** — JWT 자체 만료 검증 외 추가 안전장치

→ 검증을 여러 단계로 나눠서 하는 이유: JWT 내부 만료시간만 믿지 않고, **서버 측 데이터와 이중 대조**해서 보안성을 높임

### 컨트롤러 측 처리

```java
@PostMapping("/refresh")
public ResponseEntity<ApiResponse<TokenResponse>> refreshToken(
        @CookieValue(name="refreshToken", required = false) String refreshToken
){
    if(refreshToken == null){
        return ResponseEntity.status(HttpStatus.UNAUTHORIZED).build();
    }
    TokenResponse tokenResponse = authService.refreshToken(refreshToken);
    return buildTokenResponse(tokenResponse);
}
```

- `@CookieValue`로 쿠키에서 자동으로 값 추출
- `required = false`로 설정해서 쿠키가 없을 때 예외 대신 `null` 처리 → 직접 401 분기 처리 가능

## 5. 로그아웃 로직

```java
@PostMapping("/logout")
public ResponseEntity<ApiResponse<Void>> logout(
        @CookieValue(name="refreshToken", required = false) String refreshToken
){
    if(refreshToken != null){
        authService.logout(refreshToken);
    }

    ResponseCookie deleteCookie = createDeleteRefreshTokenCookie();
    return ResponseEntity.ok()
            .header(HttpHeaders.SET_COOKIE, deleteCookie.toString())
            .body(ApiResponse.success(null));
}
```

```java
private ResponseCookie createDeleteRefreshTokenCookie() {
    return ResponseCookie.from("refreshToken", "")
            .httpOnly(true)
            .path("/")
            .maxAge(0)     // 즉시 만료
            .sameSite("Strict")
            .build();
}
```

### 배운 점
- 로그아웃 = (1) **서버 DB에서 refreshToken 삭제** + (2) **클라이언트 쿠키도 삭제**, 두 가지가 다 필요
- 쿠키 삭제는 실제로 "지우는" 게 아니라, **같은 이름의 쿠키를 빈 값 + `maxAge(0)`으로 다시 보내서 브라우저가 즉시 만료 처리**하게 만드는 방식
- `path`, `sameSite` 등 속성을 생성할 때와 동일하게 맞춰야 브라우저가 같은 쿠키로 인식하고 덮어씀 (속성이 다르면 별개 쿠키로 인식되어 삭제가 안 될 수 있음)

## 정리 및 느낀 점

- JWT 인증 구조에서 **accessToken(짧은 생명주기, 헤더 전달)** vs **refreshToken(긴 생명주기, HttpOnly 쿠키 + DB 저장)** 으로 역할을 분리하는 이유를 코드 레벨로 직접 확인
- 보안은 한 가지 장치만으로 끝나지 않고, **JWT 서명 검증 + DB 대조 + 쿠키 속성(HttpOnly, SameSite)** 등 여러 겹으로 쌓는다는 걸 체감
- 로그인/리프레시/로그아웃 세 가지 흐름이 결국 "refreshToken을 어떻게 관리하느냐"의 변주라는 걸 이해함
