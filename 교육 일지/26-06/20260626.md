# 1. AWS RDS DB 주소 구조

## 학습 내용

* AWS RDS의 Endpoint와 JDBC URL 구조를 학습하였다.
* JDBC URL은 **프로토콜 + Endpoint + Port + Database명**으로 구성된다.
* RDS 콘솔의 **연결 및 보안(Connection & Security)** 메뉴에서 Endpoint와 Port를 확인할 수 있다.

### 예시

```text
jdbc:postgresql://naengpadb.cjc40m80uk25.ap-northeast-2.rds.amazonaws.com:5432/naengpa_db
```

| 구성                        | 설명                     |
| ------------------------- | ---------------------- |
| jdbc:postgresql://        | PostgreSQL JDBC Driver |
| naengpadb...amazonaws.com | RDS Endpoint           |
| 5432                      | PostgreSQL Port        |
| naengpa_db                | Database 이름            |

# 2. Spring Boot 프로젝트 환경 변수(env) 설정

## 학습 내용

* DB 계정, 비밀번호, JWT Secret 등 민감한 정보는 프로젝트에 직접 작성하지 않고 Environment Variable로 관리한다.
* `application.properties`에는 환경 변수(Key)만 작성하고 실제 값(Value)은 각 개발자의 환경에서 설정한다.
* 환경 변수 파일은 Git으로 관리하지 않도록 하여 보안을 유지한다.

### 예시

```properties
spring.datasource.url=${DB_URL}
spring.datasource.username=${DB_USERNAME}
spring.datasource.password=${DB_PASSWORD}
```

```env
DB_URL=jdbc:postgresql://...
DB_USERNAME=naengpa
DB_PASSWORD=*******
```

# 3. 프로젝트 코드 컨벤션

## DTO 작성 규칙

* Request, Response DTO는 Java `record`를 사용한다.
* DTO 생성 방식은 팀원 자율(`new`, `from()`, Builder 등)로 하되 하나의 클래스에서는 동일한 방식을 유지한다.

## 환경 변수 관리

* `application.yml`을 사용하지 않고 Environment Variable을 사용한다.
* 환경 변수의 Key만 프로젝트에서 공유하고 Value는 각 개발자가 개별 설정한다.

## 개발 방식

* 더 좋은 개발 방법이나 코드 작성 방식이 있다면 자유롭게 팀원과 공유한다.
* 코드 컨벤션을 크게 벗어나지 않는 범위에서 각자의 개발 방식을 존중한다.

# 4. Git 브랜치 전략

## 브랜치 생성 규칙

브랜치는 아래 규칙에 따라 생성한다.

```text
[타입]/[BE 또는 FE]-[기능명]
```

### 예시

```text
feature/BE-ingredient-register
feature/BE-score
feature/FE-login
fix/BE-inquiry
```

### 타입

* `feature` : 신규 기능 개발
* `fix` : 버그 수정
* `refactor` : 리팩토링
* `docs` : 문서 수정

# 5. 데이터베이스 테이블 및 테스트 데이터 구성

## 학습 내용

* PostgreSQL 테이블 구조와 컬럼을 확인하였다.
* 개발 및 테스트를 위해 SQL을 이용하여 직접 데이터를 INSERT하는 방법을 학습하였다.
* 회원 API가 아직 구현되지 않은 상태에서도 테스트 데이터를 활용하여 기능 개발을 진행할 수 있음을 확인하였다.
* 점수 계산을 위한 `maintenance_period` 컬럼을 추가하여 유지 기간을 저장하도록 데이터 구조를 수정하였다.

### 예시

```sql
INSERT INTO members (
    email,
    password,
    nickname,
    household_type,
    maintenance_period
)
VALUES (
    'test@test.com',
    'naeng@pa1',
    '테스트회원',
    '1인',
    0
);
```

# 💡 느낀 점

AWS RDS와 Spring Boot를 연동하는 과정에서 환경 변수 관리의 중요성을 이해할 수 있었다. 또한 프로젝트에서는 회원 기능이 아직 완성되지 않아도 테스트 데이터를 활용하여 다른 기능을 먼저 개발할 수 있다는 점을 경험하였다. ERD와 실제 DDL을 비교하면서 누락된 컬럼을 확인하는 과정도 데이터베이스 설계와 구현을 일치시키는 데 중요한 작업이라는 것을 배울 수 있었다.
