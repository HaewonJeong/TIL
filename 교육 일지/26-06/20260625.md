### 상태 기준

| 상태 | 의미 |
|---|---|
| 예정 | 아직 작업을 시작하지 않음 |
| 진행중 | 현재 작업 중 |
| 완료 | 구현 및 확인 완료 |
| 지연 | 예정된 종료일을 넘김 |
| 보류 | 정책 또는 기술 문제로 잠시 중단 |

### 우선순위 기준 (MoSCoW)

| 우선순위 | 의미 |
|---|---|
| Must | 프로젝트 핵심 기능으로 반드시 구현 |
| Should | 가능하면 구현해야 하는 주요 기능 |
| Could | 시간이 남으면 구현할 부가 기능 |
| Won't | 이번 프로젝트 범위에서는 제외 |

→ "Must 기능이 끝나기 전에는 Could 기능을 먼저 진행하지 않는다"는 규칙을 정해, 일정이 빠듯해져도 무엇을 먼저 포기할지 미리 합의해둠.

## 2. WBS → GitHub Issue로 연결

WBS에서 쪼갠 작업 단위를 그대로 GitHub Issue 관리에 적용함.

### 이슈 제목 형식

```
[도메인] 작업 내용
```
예: `[냉장고] 냉장고 및 재료 관리 기능 구현`

### Issue 관리 기준

- WBS 기준으로 **기능 단위**로 Issue 생성
- 세부 WBS 작업들은 하나의 Issue로 통합 관리
  - 예) 점수 정책 구현, 점수 계산 서비스, 점수 이력 저장 → `[냉파점수] 냉파점수 기능 구현` 이슈 하나로 묶기
- Issue 생성 시 **담당자(Assignee) 지정**, **Milestone 연결** 필수
- 진행 순서: 개발 → PR 생성 → 코드 리뷰 → Merge → Issue Close

### 이슈 본문 템플릿

**feat / refactor / docs**
```markdown
## 작업 목표
- 구현할 내용을 bullet point로 작성

## 완료 조건 (Definition of Done)
- [ ] 체크리스트 형식으로 완료 기준 명시
- [ ] 예: 재료 등록 API 응답 200 확인
- [ ] 예: 예외 처리 (중복 재료, 유효하지 않은 날짜)

## 참고 문서
- API 명세서, ERD, 기획서 링크 등
```

**bug**
```markdown
## 버그 설명
어떤 문제가 발생했는지 간략히 설명

## 재현 방법
1.
2.

## 기대 결과 / 실제 결과

## 환경
- BE / FE 구분
- 관련 브랜치:
```

**discuss**
```markdown
## 논의 주제
## 선택지
- A안 / B안
## 결정 기한
## 참여자
```

### 이슈 라벨 기준

| 라벨 | 색상 | 의미 |
|---|---|---|
| `feat` | 🟢 초록 | 새 기능 구현 |
| `bug` | 🔴 빨강 | 버그 수정 |
| `refactor` | 🔵 파랑 | 코드 개선 |
| `docs` | 🟡 노랑 | 문서 작업 |
| `discuss` | 🟣 보라 | 논의/결정 필요 |
| `blocked` | ⚫ 검정 | 진행 불가 상태 |
| `BE` | 🟠 주황 | 백엔드 작업 |
| `FE` | 🩵 하늘 | 프론트엔드 작업 |

### 칸반보드 컬럼 규칙

| 컬럼 | 이동 시점 |
|---|---|
| Todo | 이번 스프린트에 하기로 확정된 작업 |
| In Progress | 지금 실제로 작업 중인 것만 |
| Done | PR 머지 + 기능 확인까지 완료 |
| Blocked | 다른 작업/협업/API·DB 의존으로 막힘 |

## 3. 작성 시 공통 주의사항

- 작업명(이슈 제목)만 보고 내용을 파악할 수 있어야 한다
- 작업/이슈 기간은 가능하면 1~2일을 넘기지 않는다
- 담당자는 반드시 1명 이상 지정한다
- 버그 이슈는 재현 방법을 반드시 포함한다
- 기획 변경 발생 시 관련 이슈와 문서를 함께 수정한다
- 해결된 이슈는 완료일과 처리 내용을 코멘트로 남기고 닫는다
- 지연되는 작업은 회의록/이슈 문서에 원인과 대응 방안을 기록하고, 진행 불가 시 즉시 `Blocked` 처리한다
