## 3. DailyScoreScheduler 코드 분석

```java
List<Member> members = memberRepository.findAllByStatusAndDeletedAtIsNull(MemberStatus.ACTIVE);
```

- 매일 정해진 시각(cron)에 전체 활성 회원을 순회하며 처리
- 회원별로 만료된 냉장고 아이템 조회 → 있으면 만료이력 적재 + 감점, 없으면 유지기간 증가 후 4일마다 가점

### 적재/갱신 위치 정리

| 테이블 | 동작 방식 | 코드 위치 |
|---|---|---|
| `ExpiredProduct` | insert (새 row) | `run()` 메서드 본문에 직접 `save()` 호출 |
| `ScoreHistory` | insert (새 row) | `addScore()` 메서드 내부에 `save()` 호출 (간접 호출이라 run()만 보면 안 보임) |
| `Score` | update (기존 row 갱신) | `score.updateScore(newScore)` → JPA dirty checking으로 자동 UPDATE 쿼리 발생, 별도 save() 호출 없음 |

### 핵심 포인트
- `ScoreHistory` 저장이 `run()`에 안 보이는 이유: private 메서드 `addScore()` 안에 캡슐화되어 있어서 한 단계 더 들어가야 보임
- `Score`는 회원당 1개만 존재하는 "현재 점수" 레코드라 매번 새로 만들지 않고 기존 값을 찾아 update
- `ScoreHistory`는 점수 변경 이력을 누적 기록하는 로그성 테이블이라 매번 insert

### 로그 추가 계획
- 만료이력 적재: `expiredProductRepository.save()` 직후, memberId/productId/productName 포함
- 점수 계산: `addScore()` 내부, before/after 점수 값 비교 로그
- 점수이력 적재: `scoreHistoryRepository.save()` 직후, memberId/reason/targetType/targetId 포함

## 오늘의 배움 정리
- JPA 변경 감지(dirty checking): 트랜잭션 안에서 엔티티 필드만 바꿔도 커밋 시점에 자동 UPDATE 쿼리가 나간다는 점
- save(insert)와 update(dirty checking)를 구분해서 로그/리뷰 시 데이터 흐름을 명확히 추적하는 습관
- 선행 작업(탈퇴 기능)이 안 끝났어도, 필드/enum이 이미 있다면 그 기준으로 후속 작업(조회 메서드)을 먼저 진행할 수 있다는 협업 판단
- 작은 단위 변경(메서드 추가)은 굳이 PR을 쪼개지 않고 관련 기능 PR에 포함시키는 게 리뷰 효율상 낫다는 점
