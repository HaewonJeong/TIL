# 냉파 마스터
## 오늘 진행한 작업

### 1. 냉파 점수 조회 API (#93)
- `GET /api/v1/scores` 구현
- 로그인한 회원의 이메일로 회원을 조회하고, 해당 회원의 점수를 응답
- `@AuthenticationPrincipal`을 활용해 토큰에서 로그인 정보를 추출하는 방식 학습
- 예외 처리: 회원 없음(404), 점수 없음(404), 인증 실패(401)
- `ApiResponse` 공통 응답 포맷 적용

### 2. 냉파 점수 산정 내역 조회 API (#94)
- `GET /api/v1/scores/histories` 구현
- 페이징(Page) 적용 — 일반 `List` 대신 `Page` 타입을 사용해 `content`, `totalElements`, `totalPages` 등 메타정보까지 함께 응답
- 이력이 없는 경우 404가 아닌 200 + 빈 배열로 응답하도록 처리 (이력 0건은 정상 상태로 간주)

### 3. 점수/통계 스케줄러 구현 (#96, #97, #98)
- 매일 자정 실행되는 스케줄러 하나로 통합 구현 (`@Scheduled`)
- 처리 흐름:
  1. 전체 회원을 조회
  2. 회원별로 유통기한이 지난 재료(`FridgeItemService.findExpiredFridgeItems`)를 확인
  3. 만료 재료가 있으면:
     - 재료 1건당 `expired_products`에 이력 적재
     - 재료 1건당 `score_histories`에 -2점 이력 적재 및 실제 점수 차감
     - 연속 무만료 유지일수 리셋
  4. 만료 재료가 없으면:
     - 유지일수 +1
     - 유지일수가 4의 배수가 되면 +5점 가산 및 이력 적재
- 점수는 0~100점 범위로 제한(`Math.max`, `Math.min`)
- 회원 한 명 처리 중 예외가 발생해도 전체가 중단되지 않도록 `try-catch` 처리

## 배운 점
- JPA 메서드 이름 규칙(`findBy...OrderBy...`)을 잘못 쓰면 발생하는 에러 메시지 해석 방법
- `@PrePersist`로 엔티티 저장 직전 필드를 자동 채우는 방식 (DB의 `DEFAULT` 설정과는 별개로, JPA는 자바 객체 값을 그대로 사용한다는 점)
- `Page` 타입은 `.map()`을 자체적으로 제공해 페이징 정보를 유지하면서 내용만 변환 가능
- 엔티티 간 데이터 타입(Long vs int) 불일치가 실제 버그로 이어질 수 있다는 점

## 남은 과제
- `Score.gradeId`가 점수 변동 시 함께 갱신되지 않는 문제 — 등급 관리를 BE/FE 중 어디서 할지 재논의 필요
- 회원가입 시 `scores` row가 함께 생성되지 않는 케이스 발견 — 회원 도메인 담당자에게 전달 필요
