### 인터셉터
##### Filter
 - Servlet 스펙
 - **Spring MVC**에 들어오기 전 요청
 - DispatcherServlet 앞단에서 동작

##### Interceptor
- Spring MVC 기능
- **Controller** 실행 전 후 요청 처리
- Dispatcher servlet이 Controller를 찾은 뒤, Controller 호출 전 후에 동작

##### AOP
- **메서드** 실행 전후 공통 로직을 처리
- Service, Repository 등 다양한 Bean 메서드에 모두 적용 가능
- HTTP 요청이 아니어도 Spring Bean 메서드 실행 흐름에 적용 가능

```
@Configuration

public class WebConfiguration implements WebMvcConfigurer {


private final StopwatchInterceptor stopwatchInterceptor;
private final ApiKeyInterceptor apiKeyInterceptor;

//생성자 주입
//생성자가 하나만 있으면 Autowired가 생략되도 된다.
public WebConfiguration(StopwatchInterceptor stopwatchInterceptor, ApiKeyInterceptor apiKeyInterceptor){
this.stopwatchInterceptor = stopwatchInterceptor;
this.apiKeyInterceptor = apiKeyInterceptor;
}


//Spring MVC에 인터셉터 등록
@Override
public void addInterceptors(InterceptorRegistry registry) {
registry.addInterceptor(stopwatchInterceptor)
.addPathPatterns("/api/v1/**") //적용 경로
.excludePathPatterns("/error") //제외 경로
.order(1); //인터셉터 실행순서, 숫자가 작을 수록 먼저 실행
registry.addInterceptor(apiKeyInterceptor)
.addPathPatterns("/api/v1/admin/**")
.excludePathPatterns("/error")
.order(2);
}
}
```

```
이 WebConfiguration 클래스는 쉽게 말해 "내가 만든 인터셉터(문지기)들을 스프링 웹 시스템에 정식으로 등록하고, 감시할 구역과 일할 순서를 정해주는 배치 도면"입니다.

이 코드가 어떻게 가드들을 현장에 배치하는지 핵심 포인트만 쪼개서 설명해 드릴게요!

1. 🧲 문지기 가드들 소환하기 (의존성 주입)
Java

private final ApiKeyInterceptor apiKeyInterceptor;

역할: 이전에 만들어 둔 스톱워치 가드(StopwatchInterceptor)와 API Key 보안 가드(ApiKeyInterceptor)를 설정 클래스 안으로 불러오는 과정입니다.

특징: 작성하신 주석 내용 그대로, 클래스에 생성자가 딱 1개만 존재하면 스프링이 자동으로 @Autowired를 적용해 컨테이너에서 해당 객체들을 찾아 쏙 넣어줍니다(생성자 주입).

2. 🗺️ 가드들의 감시 구역과 순서 정하기 (addInterceptors)
WebMvcConfigurer 인터페이스가 제공하는 addInterceptors 메서드를 재정의(Override)하여 인터셉터를 각각 등록합니다.

① 스톱워치 인터셉터 규칙
.addPathPatterns("/api/v1/"): /api/v1/로 시작하는 모든 주소의 요청을 전부 감시(시간 측정)하겠다는 뜻입니다. (별표 두 개()는 하위 경로 전체를 포함한다는 의미)

.excludePathPatterns("/error"): 내부적으로 에러가 발생해서 /error 주소로 이동할 때는 스톱워치를 켜지 않고 제외하겠다는 뜻입니다.

.order(1): 전체 인터셉터 중에서 가장 먼저(1등으로) 실행하겠다는 규칙입니다. 숫자가 작을수록 우선순위가 높습니다.

② API Key 인터셉터 규칙
.addPathPatterns("/api/v1/admin/"): 스톱워치와 달리 이 가드는 일반 회원 주소가 아니라 오직 /api/v1/admin/으로 시작하는 관리자 전용 비밀 주소만 철저하게 감시합니다.

.order(2): 스톱워치 가드가 먼저 작동한 다음에 2등으로 바통을 이어받아 비밀번호 검사를 하겠다는 뜻입니다.

🏃‍♂️ 요청 주소에 따른 가드들의 실제 작동 흐름
이 설정 덕분에 클라이언트가 포스트맨이나 브라우저로 요청을 보낼 때, 주소에 따라 문지기들이 아래처럼 연쇄적으로 반응하게 됩니다.

💡 상황 A: 일반 회원 주소(GET /api/v1/members)로 요청할 때
order(1) 스톱워치가 튀어나와서 시간 측정을 시작합니다. (preHandle)

order(2) API Key는 주소를 보더니 "내 감시 구역(/admin/)이 아니네?" 하고 그냥 통과시켜 줍니다.

진짜 컨트롤러 메서드가 실행되어 회원 목록을 처리합니다.

모든 처리가 끝나면 스톱워치가 최종 실행 시간을 콘솔에 출력합니다. (afterCompletion)

💡 상황 B: 관리자 중요 주소(POST /api/v1/admin/secret)로 요청할 때
order(1) 스톱워치가 먼저 튀어나와서 시간 측정을 시작합니다.

이어서 order(2) API Key가 주소를 보고 슥 나타나서 헤더의 X-API-KEY를 검사합니다.

❌ 비밀번호가 틀렸다면? 401 에러 JSON을 즉시 반환하고 그 자리에서 상황 종료! 뒤에 있는 컨트롤러는 실행조차 되지 않고 차단됩니다.

⭕ 비밀번호가 맞았다면? 컨트롤러 입장 허가!
```

```
package com.ohgiraffers.interceptor;



import org.springframework.http.HttpStatus;

import org.springframework.http.ResponseEntity;

import org.springframework.web.bind.annotation.*;



import java.awt.*;

import java.util.*;

import java.util.List;



@RestController

@RequestMapping("/api/v1")

public class MenuController {



private final List<MenuDTO> menus = new ArrayList<>();



public MenuController(){

menus.add(new MenuDTO(1, "아메리카노", 3000));

menus.add(new MenuDTO(2, "카페라떼", 4000));

menus.add(new MenuDTO(3, "바닐라라떼", 4500));

}



@GetMapping("/menus")

public ResponseEntity<List<MenuDTO>> fineMenus() throws InterruptedException {

Thread.sleep(3000);

return ResponseEntity.ok(menus);

}



@GetMapping("/menus/{menuCode}")

public ResponseEntity<MenuDTO> findMenu(@PathVariable int menuCode) throws InterruptedException{

Thread.sleep(2000);



MenuDTO foundMenu = menus.stream()

.filter(menu -> menu.getMenuCode() == menuCode)

.findFirst()

.orElseThrow(() -> new IllegalArgumentException(menuCode + "번 메뉴가 존재하지 않음!") );



return ResponseEntity.ok(foundMenu);

}



@PostMapping("/admin/menus")

public ResponseEntity<Map<String, Object>> registMenu(@RequestBody MenuDTO menu){

menus.add(menu);



Map<String, Object> responds = new LinkedHashMap<>();

responds.put("message", "관리자 메뉴 등록 성공");

responds.put("menu", menu);



return ResponseEntity.status(HttpStatus.CREATED).body(responds);

}



}



package com.ohgiraffers.interceptor;



import jakarta.servlet.http.HttpServletRequest;

import jakarta.servlet.http.HttpServletResponse;

import org.springframework.http.HttpStatus;

import org.springframework.http.MediaType;

import org.springframework.stereotype.Component;

import org.springframework.web.servlet.HandlerInterceptor;

import tools.jackson.databind.ObjectMapper;



import java.util.LinkedHashMap;

import java.util.Map;



@Component

public class ApiKeyInterceptor implements HandlerInterceptor {



//관리자 API 요청에서 확인할 헤더 이름

private static final String API_KEY_HEADER = "X-API-KEY";

private static final String VALID_API_KEY = "secret-key";

//수업용 유효한 API Key 값

private final ObjectMapper objectMapper;



public ApiKeyInterceptor(ObjectMapper objectMapper){

this.objectMapper = objectMapper;

}





//preHandle: Controller 실행전에 호출, false를 반환하면 요청 중단

@Override

public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {

/*Controller에 도착하기 전에 요청 헤더에서 API Key를 꺼낸다.*/

String apiKey = request.getHeader(API_KEY_HEADER);



if(VALID_API_KEY.equals(apiKey)){

System.out.println("[ApiKey] authorized " + request.getMethod() + " " + request.getRequestURI());

return true;

}



System.out.println("[ApiKey] unauthorized " + request.getMethod() + " " + request.getRequestURI());



response.setStatus(HttpStatus.UNAUTHORIZED.value()); //401

response.setContentType(MediaType.APPLICATION_JSON_VALUE);

response.setCharacterEncoding("UTF-8");



Map<String, Object> errorResponse = new LinkedHashMap<>();

errorResponse.put("status", HttpStatus.UNAUTHORIZED.value());

errorResponse.put("error", HttpStatus.UNAUTHORIZED.name());

errorResponse.put("message", "관리자 API 호출하려면 X-API-KEY 헤더가 필요합니다.");

errorResponse.put("path", request.getRequestURI());



response.getWriter().write(objectMapper.writeValueAsString(errorResponse));



return false; //요청 흐름 중단, /admin Controller 메서드 실행



}

}

package com.ohgiraffers.interceptor;



import jakarta.annotation.Nullable;

import jakarta.servlet.http.HttpServletRequest;

import jakarta.servlet.http.HttpServletResponse;

import org.springframework.stereotype.Component;

import org.springframework.web.servlet.HandlerInterceptor;





@Component

public class StopwatchInterceptor implements HandlerInterceptor {



/*preHandle에서 저장하고 afterCompletion에서 다시 꺼낼 요청 속성 이름*/

private static final String START_TIME = "startTime";



@Override

public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {

long startTime = System.currentTimeMillis();



request.setAttribute(START_TIME, startTime); //요청 객체 안에 시작 시간을 임시 보관



System.out.println("[Stopwatch] preHandle " + request.getMethod() + " " + request.getRequestURI());



return true; //요청 게속 진행, false를 반환하면 Controller가 실행되지 않음

}



//afterCompletion: 요청 처리가 끝난 뒤 호출됨, 로그, 처리시간 측정, 정리작업에 적합

@Override

public void afterCompletion(HttpServletRequest request, HttpServletResponse response, Object handler, @Nullable Exception ex) throws Exception{



/*같은 요청 객체에 저장해둔 시작 시간을 꺼낸다.*/

Object startTimeAttribute = request.getAttribute(START_TIME);



//instanceof pattern matching 문법

//startTimeAttribute가 Long 타입이면 그 값을 Long으로 캐스팅해서 startTime 변수로 바로 사용

if(startTimeAttribute instanceof Long startTime){

long elapsedTime = System.currentTimeMillis() - startTime;



response.setHeader("X-Elapsed-Time", elapsedTime + "ms");



System.out.println("[Stopwatch] afterCompletion " + request.getMethod()

+ " " + request.getRequestURI()

+ " " + elapsedTime + "ms");

}

}

}







package com.ohgiraffers.interceptor;



import org.springframework.context.annotation.Configuration;

import org.springframework.web.servlet.config.annotation.InterceptorRegistry;

import org.springframework.web.servlet.config.annotation.WebMvcConfigurer;



@Configuration

public class WebConfiguration implements WebMvcConfigurer {



private final StopwatchInterceptor stopwatchInterceptor;

private final ApiKeyInterceptor apiKeyInterceptor;



//생성자 주입

//생성자가 하나만 있으면 Autowired가 생략되도 된다.

public WebConfiguration(StopwatchInterceptor stopwatchInterceptor, ApiKeyInterceptor apiKeyInterceptor){

this.stopwatchInterceptor = stopwatchInterceptor;

this.apiKeyInterceptor = apiKeyInterceptor;

}



//Spring MVC에 인터셉터 등록

@Override

public void addInterceptors(InterceptorRegistry registry) {

registry.addInterceptor(stopwatchInterceptor)

.addPathPatterns("/api/v1/**") //적용 경로

.excludePathPatterns("/error") //제외 경로

.order(1); //인터셉터 실행순서, 숫자가 작을 수록 먼저 실행

registry.addInterceptor(apiKeyInterceptor)

.addPathPatterns("/api/v1/admin/**")

.excludePathPatterns("/error")

.order(2);



}

}
```

```
프론트엔드나 포스트맨에서 어떤 주소로 요청을 보내느냐에 따라 이 코드들이 일하는 순서가 완전히 달라집니다. 가장 대표적인 3가지 상황으로 나누어서 컴퓨터가 코드를 실행하는 순서를 아주 명확하게 시각화해 드릴게요.

🗺️ 시작하기 전, 인터셉터 배치도 리마인드 (WebConfiguration)
1순위 (order 1): StopwatchInterceptor ➡️ /api/v1/ (모든 주소 감시)

2순위 (order 2): ApiKeyInterceptor ➡️ /api/v1/admin/ (관리자 주소만 감시)

☕ 상황 1. 일반 메뉴 전체 조회할 때
요청: GET http://localhost:8080/api/v1/menus

StopwatchInterceptor.preHandle() 실행

현재 시간(startTime)을 측정해서 request 주머니에 임시 보관합니다.

콘솔 출력: [Stopwatch] preHandle GET /api/v1/menus

return true; 다음 단계로 패스!

ApiKeyInterceptor.preHandle() 확인

WebConfiguration 설정상 이 인터셉터는 /admin/ 경로만 감시하게 되어 있습니다.

지금 주소는 일반 메뉴 조회(- /menus)이므로 그냥 아무 일도 안 하고 통과합니다.

MenuController.fineMenus() 실행 (컨트롤러)

코드를 보면 Thread.sleep(3000);이 적혀 있네요! 서버가 여기서 3초 동안 얌전히 대기(잠들기)합니다.

3초가 지나면 메뉴 리스트(menus)를 200 OK 상태 코드로 담아 응답 상자를 만듭니다.

StopwatchInterceptor.afterCompletion() 실행

프론트엔드로 응답이 나가기 직전, request 주머니에서 처음에 넣어둔 startTime을 꺼냅니다.

3초 동안 잠들었기 때문에 계산된 elapsedTime은 약 3000ms 전후가 나옵니다.

프론트엔드가 볼 수 있게 응답 헤더에 담아줍니다: response.setHeader("X-Elapsed-Time", "...ms");

콘솔 출력: [Stopwatch] afterCompletion GET /api/v1/menus 3005ms

🔐 상황 2. 관리자가 올바른 키를 넣고 메뉴를 등록할 때
요청: POST http://localhost:8080/api/v1/admin/menus
헤더: X-API-KEY: secret-key (정상 비밀번호 포함)

StopwatchInterceptor.preHandle() 실행

시작 시간 측정 후 request에 보관.

콘솔 출력: [Stopwatch] preHandle POST /api/v1/admin/menus

return true; 패스!

ApiKeyInterceptor.preHandle() 실행 (이번엔 주소가 /admin/이라서 작동!)

헤더에서 X-API-KEY 값을 꺼내왔더니 우리가 정해둔 "secret-key"와 일치합니다!

콘솔 출력: [ApiKey] authorized POST /api/v1/admin/menus

return true; 다음 진짜 컨트롤러로 패스!

MenuController.registMenu() 실행 (컨트롤러)

보낸 메뉴 데이터를 리스트에 추가하고 201 Created 성공 메시지 상자를 만듭니다.

StopwatchInterceptor.afterCompletion() 실행

이 메서드는 Thread.sleep이 없으므로 매우 빠르게 처리됩니다. (예: 5ms)

응답 헤더에 가공된 시간을 추가해 줍니다.

콘솔 출력: [Stopwatch] afterCompletion POST /api/v1/admin/menus 5ms

🚨 상황 3. 해커나 일반 유저가 키 없이 관리자 메뉴 등록을 시도할 때 (입구 컷)
요청: POST http://localhost:8080/api/v1/admin/menus
헤더: X-API-KEY 헤더를 안 보냈거나 틀린 값을 보냄

StopwatchInterceptor.preHandle() 실행

어김없이 첫 번째 형님 문지기가 시작 시간을 재기 시작합니다.

콘솔 출력: [Stopwatch] preHandle POST /api/v1/admin/menus

return true; 패스!

ApiKeyInterceptor.preHandle() 실행

헤더를 뒤졌는데 비밀번호가 없거나 틀렸습니다! 인증 실패 로직 가동!

콘솔 출력: [ApiKey] unauthorized POST /api/v1/admin/menus

response 상자에 직접 401 Unauthorized 상태와 JSON 형식을 주입하고, objectMapper로 에러 메시지 바디를 직접 그려서 채워 넣습니다.

return false;를 던지며 전체 흐름을 강제로 중단(Short-circuit)시킵니다.

🛑 MenuController 실행 스킵!

인터셉터가 문을 잠가버렸기 때문에, 뒤에 있는 MenuController.registMenu()는 구경도 못 하고 실행되지 않습니다. (서버 자원 보호 완료!)

StopwatchInterceptor.afterCompletion() 실행

주의할 점: 컨트롤러는 안 갔지만, 1단계에서 스톱워치가 켜졌고 요청이 끝난 건 맞으므로 afterCompletion은 무조건 실행됩니다!

입구 컷 당할 때까지 걸린 찰나의 시간(예: 2ms)을 계산합니다.

콘솔 출력: [Stopwatch] afterCompletion POST /api/v1/admin/menus 2ms

📝 핵심 요약 정리
필터링 패턴: 주소 매칭 규칙에 따라 Stopwatch는 언제나 기본 베이스로 작동하고, /admin/ 주소일 때만 ApiKey 가드가 2등으로 합세합니다.

성공 흐름: Stopwatch(전) ➡️ ApiKey(전) ➡️ 컨트롤러 ➡️ Stopwatch(끝)
```

```
@NotNull: null만 허용 안 함. (""이나 " "은 통과됨)

@Min / @Max: 숫자형 데이터의 최소/최대값 지정.

@Pattern(regexp = "..."): 정규표현식을 써서 비밀번호 형식(영문+숫자+특수문자 조합 등)이나 전화번호 형식을 칼같이 검사할 때 사용.
```

### Swagger 라이브러리
- SpringDoc OpenAPI : OpenApi 문서화, Swagger
- implementation 'org.springdoc:springdoc-openapi-starter-webmvc-ui:3.0.2'
 - 스프링 컨트롤러 코드와 스웨거로 문서를 자동 생성해준다.
- SpringDoc OpenAPI의 작동 원리
```
1. 스캔: 스프링 부트 서버가 켜질 때, springdoc 라이브러리가 프로젝트 내부의 @RestController를 샅샅이 탐색합니다.
2 JSON 생성: 매핑된 URL 주소, DTO 구조, Validation 규칙(@NotBlank, @Past 등)을 수집해서 하나의 거대한 OpenAPI 표준 규격 JSON 문서를 메모리에 생성합니다.
3. 렌더링: 내장된 Swagger UI 웹페이지가 그 JSON 문서를 읽어서 우리가 브라우저로 보는 그래픽 화면으로 예쁘게 그려주는 원리입니다! 코드가 바뀌면 서버가 재시작될 때 JSON도 새로 생성되므로 문서가 항상 100% 최신 상태를 유지하게 됩니다.
```
#### Swagger 설정 및 사용방법

##### Swagger란?
- Swagger는 RESTful API를 설계하고 문서화하는데 사용되는 오픈 소스 프레임워크입니다.
- Swagger는 API의 구조를 정의하는데 사용할 수 있는 여러 도구와 라이브러리를 제공합니다. 이를 통해 API의 구조를 정의하는데 사용할 수 있는 여러 도구와 라이브러리를 제공합니다. 이를 통해 API 문서를 자동으로 생성하고, API를 테스트하는 인터페이스를 제공할 수 있습니다.

##### 장점
- 자동화된 문서화
 - Swagger는 API문서를 자동으로 생성해주기 때문에 개발자가 일일이 문서를 작설할 필요가 없습니다.
- 시간적 인터페이스
 - Swagger UI를 통해 API를 시각적으로 테스트하고 탐색할 수 있습니다.
- 표준화
 - Swagger는 OpenAPI Specification(OAS)을 사용하여 API 문서화를 표준화합니다. 이를 통해 다른 팀이나 회사에서도 쉽게 이해하고 사용할 수 있습니다.
- 코드 생성
 - Swagger는 클라이언트 SDK와 서버 스텁 코드를 자동으로 생성할 수 있습니다.
- API 테스트
 - Swagger UI를 통해 API를 직접 호출하고 테스트 할 수 있습니다.
##### 단점
- 초기 설정
 - 프로젝트에서 Swagger를 설정하고 통합하는 초기 과정이 다소 복잡할 수 있습니다.
- 오버헤드
 - Swagger를 사용하면 API 문서화와 관련된 추가 코드와 설정이 필요해지기 때문에 프로젝트 규모가 커질수록 관리가 복잡해질 수 있습니다.
- 실시간 반영 제한
  - 코드 변경 사항이 항상 실시간으로 문서화에 반영되지 않을 수 있습니다. 주기적으로 문서를 업데이트해야 할 필요가 있습니다.

#### Controller
- 컨트롤러에서 Swagger 어노테이션을 사용하는 방법입니다.

 - 메서드의 설명, 요청/응답 모델, 파라미터 등을 지정할 수 있습니다.

#### DTO
- @Schema
 - Swagger에서 API 요청/응답 모델의 스키마 정보를 문서화하는 데 사용되는 주석(Annotation)입니다.
- @Schema의 주요 속성은 다음과 같습니다
 - name: 모델의 이름을 지정합니다.
 - description: 모델에 대한 설명을 제공합니다.
 - required: 필수 필드를 지정합니다.
 - properties: 모델의 필드 정보를 제공합니다.
 - implementation: 모델을 구현하는 클래스를 지정합니다.
