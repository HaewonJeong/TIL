3. 멀티 스레드(Multi-Thread)와 웹 서버(Tomcat)의 마법
"여러 요청이 동시에 들어오면 여러 스레드가 작업을 할 수 있다." 이 구절이 바로 혜원님이 방금 전 공부하신 서블릿(Servlet)과 톰캣이 위대한 이유입니다.
만약 일꾼이 단 한 명인 싱글 스레드 서버라면 어떻게 될까요?
사용자 A가 서버에 접속해서 영화 예매 티켓을 조회합니다. (일꾼이 A를 돕느라 바쁨)

이때 사용자 B가 접속하면, 일꾼이 한 명뿐이라 A의 작업이 끝날 때까지 B는 하얀 화면을 보며 무한 대기를 해야 합니다. 엄청 답답하겠죠?

- 하지만 우리가 쓰는 톰캣은 멀티 스레드(Multi-Thread) 방식으로 작동합니다.
톰캣은 내부에 일꾼(스레드)들을 미리 수십~수백 명 고용해 둡니다. (스레드 풀, Thread Pool)

사용자 A가 들어오면 1번 일꾼을 배정해서 아까 만든 서블릿의 service() 메서드를 실행하게 합니다.

그와 동시에 사용자 B가 들어오면, 기다리게 하지 않고 2번 일꾼을 새로 붙여서 똑같이 service()를 실행합니다.

덕분에 수천 명의 사용자가 동시에 "장르별 영화 보여줘!"라고 요청해도, 각각의 일꾼(스레드)들이 동시다발적으로 service()를 쿵짝쿵짝 실행하기 때문에 아무도 기다리지 않고 빠른 서비스를 받을 수 있는 거랍니다.

1. 자바의 아버지, 썬 마이크로시스템즈 (Sun)
원래 자바는 1995년 Sun이라는 회사에서 제임스 고슬링이라는 개발자(자바의 아버지)의 손에 의해 태어났습니다.

그 당시 Sun 회사의 개발자들이 자바 가상머신(JVM)의 내부 깊은 곳(엔진 내부 속성이나 특수한 기능들)을 개발하면서, 자신들이 만든 고유 영역이라는 것을 표시하기 위해 패키지나 속성 이름 앞에 sun.* 이라는 접두사를 붙이기 시작했어요.

2. 오라클(Oracle)의 인수, 하지만 남겨진 흔적
세월이 흘러 2010년에 오라클(Oracle)이라는 대기업이 Sun 마이크로시스템즈를 통째로 인수하게 됩니다. 이때부터 자바의 주인은 오라클이 되었죠.

주인은 바뀌었지만, 자바 엔진(JVM) 내부 깊숙한 곳에 박혀있는 수천, 수만 개의 코드들을 일일이 oracle.stdout...으로 바꾸기에는 리스크가 너무 컸습니다. 이름을 함부로 바꿨다가 전 세계에서 돌아가고 있는 수억 개의 자바 프로그램들이 "어? sun으로 시작하는 속성이 어디 갔지?" 하면서 다 뻗어버릴 테니까요! 🥶

그래서 전 세계 자바 개발자들의 평화를 위해 "주인은 오라클이지만, 내부 엔진 속성 이름은 최초 창조주였던 sun 이름을 그대로 남겨두자!" 하고 약속한 것입니다.

3. 우리가 쓰는 sun.stdout 의 진짜 의미
결국 우리가 입력한 옵션은 컴퓨터에게 이렇게 말하는 것과 같습니다.

"자바(JVM) 내부 엔진(sun)아, 네가 관리하는 기본 출력창(stdout)의 글자 규격(encoding)을 UTF-8로 세팅해 줘!"

#### Sevrlet 동작 방식
- 사용자(클라이언트)가 URL을 입력하면 HTTP Request가 Servlet Container로 전송합니다.
- 요청을 전송받은 Servlet Container는 HttpServletRequest, HttpServletResponse 객체를 생성합니다.
- web.xml을 기반으로 사용자가 요청한 URL이 어느 서블릿에 대한 요청인지 찾습니다.
- 해당 서블릿에서 service메소드를 호출한 후 클리아언트의 GET, POST여부에 따라 doGet() 또는 doPost()를 호출합니다.
- doGet() or doPost() 메소드는 동적 페이지를 생성한 후 HttpServletResponse객체에 응답을 보냅니다.
- 응답이 끝나면 HttpServletRequest, HttpServletResponse 두 객체를 소멸시킵니다.
