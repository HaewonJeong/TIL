### 예외 처리
#### throws Exception : 예외처리를 호출하는 시점으로 위임" 한다는 의미
```
public static void main(String[] args) throws Exception{
        //예외처리를 나를 호출하는 시점으로 위임.
        //JVM까지 가서 프로그램이 비정상 종료 처리 된다.
        ExceptionTest et = new ExceptionTest();

        System.out.println("정상 케이스");
        et.checkEnoughMoney(10000,50000);

        System.out.println("프로그램을 종료합니다.");
    }
```

```
1. "예외처리를 호출하는 시점으로 위임" 한다는 의미
코드에서 throws Exception을 붙이는 행위를 말합니다.

원래 방식 (try-catch): checkEnoughMoney를 실행하다가 돈이 부족하면 그 자리에서 즉시 "잔액이 부족합니다"라고 수습하는 방식입니다.

위임 방식 (throws): checkEnoughMoney는 "난 돈이 부족한지 확인만 할 뿐이야. 부족해서 생기는 문제는 나를 호출한 main 네가 알아서 처리해!"라고 던져버리는 것입니다.

그런데 지금 main 메서드마저도 throws Exception을 쓰고 있죠? 이건 main도 책임을 지지 않고 자신을 호출한 존재에게 다시 던지겠다는 뜻입니다.

2. "JVM까지 가서 비정상 종료 처리" 된다는 의미
자바 프로그램을 실행하는 가장 윗순위인 JVM(자바 가상 머신)은 main 메서드를 호출하는 주체입니다.

checkEnoughMoney에서 예외 발생 → main으로 던짐.

main도 수습(try-catch) 안 하고 throws로 던짐.

결국 최종 보스인 JVM에게 예외가 전달됩니다.

JVM은 예외를 받으면 다음과 같이 행동합니다.

콘솔창에 에러 메시지(Stack Trace)를 붉은 글씨로 출력합니다.

"더 이상 처리해줄 사람이 없네?"라고 판단하고 프로그램을 강제로 즉시 종료시킵니다.

3. 실생활 비유
이 상황을 회사 업무로 비유하면 이해가 빠릅니다.

사원(checkEnoughMoney): "부장님, 돈이 없어서 결제가 안 됩니다!" (예외 발생) -> "저는 권한이 없으니 부장님이 처리해 주세요." (throws)

부장(main): "나도 처리하기 귀찮은데? 사장님한테 보고해야지." (throws)

사장(JVM): "아무도 이 문제를 해결 안 해? 그럼 이 프로젝트(프로그램) 당장 때려쳐(종료)!"
```

#### 예외처리 메서드
- e.getMessage() : 에러의 원인을 간단하게 출력
- e.printStackTrace() : 예외 발생 시 예외 메시지와 함께 호출 스택 정보를 출력
- e.getClass() : 에러 발생 클래스

#### 멀티 캐치 문법
```
try {
            et.checkEnoughMoney(-3000, 10000);
            //multi-catch : 여러 예외 타입을 | 기호로 묶어 하나의 catch 블록에서 처리 가능
        } catch (PriceNegativeException | MoneyNegativeException e) {
            System.out.println("진단명: "+e.getClass() + "발생!!"+ e.getMessage());
        }
```
- 부모의 예외 처리 클래스 보다 더 넓은 범위의 예외처리를 던질 수 없다.
- throw : 실제 예외 던짐
- throws : 책임 전가(위임)
- try-catch : 
- 커스텀 예외 : extends 예외 클래스
- catch 순서 : 자식 > 부모순으로 위에서 아래로 catch 블럭 작성
- 오버라이딩 : 같거나 더 구체적으로만, 더 넓게는 불가 하다.
#### Checked Exception과 Unchecked Exception
Unchecked Exception
- 런타임 시점에 발생하는 예외로, 컴파일러가 처리 여부를 강제하지 않습니다. 자바에서는 RuntimeException을 상속한 예외들이 해당됩니다.
- 일반적으로 프로그래머의 실수나 코드 오류로 인해 발생합니다.
- ex) null인 참조변수의 멤버호출(NullPointException), 배열의 범위를 벗어난 동작(ArrayIndexOutOfBoundException) 등
Checked Exception
- Checked Exception을 유발하는 메서드를 호출하는 경우, 메서드 시그니처에 throws를 사용하여 호출자에게 예외를 위임하거나 메서드 내에서 try-catch를 사용하여 해당 예외를 반드시 처리해야합니다.
- Checked Exception은 컴파일 시점에 확인되며, 반드시 처리해야 하는 예외입니다. 자바에서는 IOException, SQLException 등이 이에 속합니다.
- ex) 존재하지 않는 파일의 입력(FileNotFoundException), 클래스 이름을 잘못 입력(ClassNotFoundException) 등

Error와 Exception의 차이는 무엇인가요?
- Error는 주로 JVM에서 발생하는 심각한 문제로, OutOfMemoryError, StackOverflowError 등 시스템 레벨에서 발생하는 오류입니다. 이는 일반적으로 프로그램에서 처리하지 않으며, 회복이 어려운 오류에 속하며, 애플리케이션 코드에서 복구할 수 없는 심각한 문제를 나타냅니다.
- 반면, Exception은 프로그램 실행 중 발생할 수 있는 오류 상황을 나타냅니다. 대부분의 경우 회복 가능성이 있으며, 프로그램 내에서 예외 처리를 통해 오류 상황을 제어할 수 있습니다. 
- Exception은 다시 Checked Exception과 Unchecked Exception으로 나눌 수 있습니다.
