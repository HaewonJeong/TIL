# 개발
## 1. SecurityConfig 인증 경로 수정

- 기존 경로(`/api/v1/naengpa-scores`, `/api/v1/naengpa-score/**`, `/api/v1/naengpa-stats/**`)가 실제 구현 경로와 불일치
- 실제 API 명세 기준으로 경로 수정
  ```java
  "/api/v1/scores",
  "/api/v1/scores/**",
  "/api/v1/member-stats/**"
  ```
- `/api/v1/scores`와 `/api/v1/scores/**`를 별도로 선언한 이유: `/**`는 하위 경로만 잡고 정확히 `/api/v1/scores` 자체는 못 잡기 때문

## 2. MemberRepository 활성 회원 조회 메서드 추가

- 스케줄러에서 탈퇴/비활성화 회원 제외 처리를 위해 메서드 추가
  ```java
  List<Member> findAllByStatusAndDeletedAtIsNull(MemberStatus status);
  ```
- 탈퇴 기능 자체는 아직 미구현이나, `MemberStatus` enum(`ACTIVE`, `INACTIVE`)은 이미 존재
- 관리자가 회원 비활성화 시 `status = INACTIVE`로 변경되는 로직 확인

## 3. DailyScoreScheduler 로그 추가

- 기존 로그가 식별자 없이 동작 여부만 출력하는 수준이었음
- 세 군데로 구분하여 로그 추가
  - 만료 이력 적재: `expiredProductRepository.save()` 직후, `memberId` / `productId` / `productName` 포함
  - 점수 계산 및 갱신: `addScore()` 내부, before/after 점수 포함
  - 점수 이력 적재: `scoreHistoryRepository.save()` 직후, `memberId` / `reason` / `targetType` / `targetId` 포함

## 정리
- `/**` 패턴은 하위 경로만 잡기 때문에 루트 경로 자체는 별도로 등록해야 한다
- JPA dirty checking: `@Transactional` 안에서 엔티티 필드 변경만 해도 커밋 시 자동 UPDATE 발생 (별도 `save()` 불필요)
- 이력성 테이블(ScoreHistory)은 insert, 현재값 테이블(Score)은 update — 역할에 따라 저장 방식이 다름

# 브랜치 명령어
## 브랜치 생성 및 이동

```bash
# dev 브랜치로 이동
git checkout dev

# dev 최신화
git pull origin dev

# 새 브랜치 생성 + 이동 (한 번에)
git checkout -b feature/브랜치이름
```

## 브랜치 목록 확인

```bash
# 로컬 브랜치 목록 (현재 브랜치에 * 표시)
git branch
```

## 브랜치 삭제

```bash
# 안전하게 삭제 (merge된 브랜치만 삭제됨)
git branch -d 브랜치이름

# 여러 개 한 번에 삭제
git branch -d 브랜치1 브랜치2 브랜치3

# 강제 삭제 (merge 여부 무관)
git branch -D 브랜치이름
```

## 원격(GitHub)에서 삭제된 브랜치 로컬에 동기화

```bash
# GitHub에서 삭제된 브랜치 추적 정보 로컬에서 정리
git fetch --prune
```

> 주의: `--prune`은 원격 추적 정보만 정리하고, 로컬 브랜치 자체는 직접 `git branch -d`로 따로 삭제해야 함

## 작업 중인 변경사항 버리기

```bash
# 스테이징 해제 (add 취소)
git restore --staged .

# 워킹 디렉토리 변경사항 원상복구
git restore .

# git이 추적하지 않는 새 파일/폴더 강제 삭제
git clean -fd
```

> 주의: `restore`와 `clean -fd`는 되돌릴 수 없음. 실행 전 `git status`로 반드시 확인

## 현재 브랜치를 원격 dev와 동일하게 맞추기

```bash
git fetch origin
git reset --hard origin/dev
```

> 주의: 커밋 안 한 모든 변경사항이 완전히 삭제됨

## add 없이 폴더 단위로 한 번에 올리기

```bash
# 폴더 단위로 add (끝에 슬래시 필수)
git add src/main/java/com/naengpa/naengpamasterbackend/score/
git add src/main/java/com/naengpa/naengpamasterbackend/statistics/
```

> env 파일 실수 방지를 위해 `git add .` 대신 폴더/파일 단위로 직접 지정

## 전체 흐름 (PR 올리는 순서)

```bash
# 1. dev 최신화
git checkout dev
git pull origin dev

# 2. 작업 브랜치 생성
git checkout -b feature/브랜치이름

# 3. 코드 작업 후 add (폴더/파일 단위로)
git add 경로

# 4. 커밋
git commit -m "feat: 작업 내용 요약 (#이슈번호)"

# 5. push
git push origin feature/브랜치이름

# 6. GitHub에서 PR 생성
```

## 정리

- 커밋 메시지는 50자 이내로 작성하는 것이 관례
- `git status`는 add/커밋 전에 항상 확인하는 습관 들이기
- `application-secret.env`는 절대 add하면 안 됨 → 폴더 단위 add로 방지
- `git branch -d`로 안 지워지면 이미 merge 안 된 브랜치 → `-D`로 강제 삭제
- GitHub에서 브랜치를 지워도, 로컬에는 자동으로 반영 안 됨 → `git fetch --prune` + `git branch -d` 필요
