## 상세 내용

### 1. 통계 모달 (StatsDetailModal) API 연동
- 세 가지 통계 API를 순서대로 연동
  - `GET /api/v1/member-stats/top-ingredients?days=7` — 가장 많이 만료된 재료 TOP5
  - `GET /api/v1/member-stats/expired-categories?days=7` — 카테고리별 만료량
  - `GET /api/v1/member-stats/expired-records?days=7` — 최근 만료 기록
- `useState` + `useEffect` + `alive` 플래그 패턴으로 비동기 요청의 race condition을 방지하는 방식을 학습 (컴포넌트 언마운트/의존성 변경 시 이전 요청의 응답을 무시)

### 2. 트러블슈팅: 동일 요청인데 응답이 다른 문제
- 필터 버튼(1일/2일)을 눌러 테스트하던 중, 같은 파라미터로 요청해도 응답 JSON이 달라지는 현상 발견
- 원인 분석 결과, 만료 개수가 동점인 재료(예: 시금치·김치 둘 다 9개)에 대해 백엔드 쿼리에 2차 정렬 기준(tie-breaker)이 없어서 매번 순서가 달라지는 것으로 파악
- React 18 StrictMode의 개발 모드 double-invoke 동작도 함께 확인 (프로덕션 빌드에서는 발생하지 않음)
- 프론트 문제가 아니라 백엔드 쿼리의 `ORDER BY` 이슈임을 확인하고 별도 전달

### 3. 카테고리 → 이모지 매핑 구조 이해
- 앱 전역에서 쓰이는 2단계 변환 파이프라인을 파악
  1. `productCategoryId`(숫자, 백엔드) → `CATEGORY_NAMES`로 문자열 카테고리명 변환
  2. 문자열 카테고리명 → `CATEGORY_EMOJIS`로 이모지 매핑
- 재료명 기반 매핑(`PRESET_INGREDIENTS`)은 폴백(fallback)일 뿐이고, id 기반 매핑이 정석 패턴이라는 것을 확인
- 다른 담당자가 소유한 파일(`useIngredientStore.js`)의 상수를 `export`해서 재사용할지, 로컬에 복제해서 쓸지 협업 관점에서 고민 → 최종적으로 담당자 확인 후 `export`로 통일하고 중복 코드 제거

### 4. 냉파 점수(Score) API 연동
- 현재 점수: `GET /api/v1/scores`
- 점수 산정 내역(페이지네이션): `GET /api/v1/scores/histories?size=7`
- `scoreReason`(`EXPIRED_PRODUCT`, `RECIPE_CREATED`) 값에 따라 아이콘/설명 문구를 다르게 매핑하는 테이블(`SCORE_REASON_META`) 설계
- 만료 재료 항목은 카테고리 이모지를, 그 외 항목은 고정 아이콘을 쓰도록 분기 처리

### 5. 등급표 캐릭터 이미지 추가
- `GRADE_TABLE`에 이미 정의돼 있던 `characterImage` 필드를 활용해 등급표 각 칸(냉털 새내기~냉파 마스터) 위에 캐릭터 이미지를 작게 표시
- 이미지 파일명이 번호순(`1.newbie.png` ~ `6.master.png`)으로 변경되어 매핑 경로 업데이트
