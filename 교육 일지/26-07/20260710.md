## 📌 오늘 핵심 피드백

### 프로젝트는 끝난 것이 아니라 '디벨롭'하는 것이 중요하다.
현재 내가 개발한 기능 중 **스케줄러**와 **배포**는 포트폴리오에서 강조 하고 싶은 부분이다. 다만 지금 상태에서 끝내는 것이 아니라, **추가 개선(디벨롭)을 통해 경쟁력을 높이는 것이 좋다**는 피드백을 받았다.

# 🚀 내 프로젝트 디벨롭 방향

## 1. 스케줄러

현재 구현한 스케줄러를 조금 더 운영 환경에 가깝게 개선해보기

### 포트폴리오에 작성하면 좋은 내용
- try-catch를 활용하여 하나의 작업 실패가 전체 배치를 중단시키지 않도록 처리
- 예외 발생 시 로그를 남기고 후속 작업은 계속 수행
- 로그 레벨(INFO, WARN, ERROR) 정리
- 실패 원인을 쉽게 추적할 수 있도록 개선
- 예외 발생 시 전체 배치가 중단되지 않도록 설계
- 로그 기반으로 장애를 추적할 수 있도록 개선
- 운영 환경을 고려한 예외 처리 적용

---

## 2. 배포

현재 배포 경험을 운영 관점까지 확장하기

### 개선 아이디어
- 배포를 팀플에서 진행하지 않는 사람은 AWS Free Tier에서 직접 배포
- 고객에게 발생할 수 있는 장애 상황 고민
- 무중단 배포 방식 조사
- 롤백 전략 정리
- 배포 자동화(CI/CD) 과정 팀원에게 설명

### 포트폴리오에 작성하면 좋은 내용
- 실제 운영 환경을 고려한 배포 경험
- 장애 발생 시 대응 및 롤백 전략 고민
- 운영 관점에서 배포 프로세스를 개선한 경험

# ☕ Java 및 객체지향

## 프로세스와 스레드

### 프로세스
- 실행 중인 프로그램
- 예) Spring Boot 서버

### 스레드
- 프로세스 내부에서 작업을 수행하는 실행 단위
- Thread Pool을 이용하면 효율적으로 관리 가능

---

## 예외 처리

### try-catch-finally

프로젝트에서도 적극 활용하기

예시

- Scheduler 작업 실패
- 다음 작업은 계속 수행
- 로그 기록
- finally에서 자원 정리

---

## 캡슐화

- private으로 데이터 보호
- public 메서드를 통해 접근
- 객체지향 설계 이유를 설명할 수 있어야 한다.

## 인터페이스 vs 추상 클래스

### 인터페이스

- 다중 구현 가능
- 기능 명세

### 추상 클래스

- 단일 상속
- 공통 기능 구현

---

## 추상화

장점

- 사용하기 편하다.
- 코드 재사용성이 높다.

단점

- 문제가 발생하면 원인을 찾기 어렵다.

예시

- JPA
- Hibernate

적절한 추상화 수준을 고민해야 한다.
