- 일반 JOIN은 SQL 조인만 수행하고 연관 엔티티는 즉시 로딩하지 않는다. 
- Fetch Join은 연관 엔티티까지 함께 조회하여 N+1 문제를 해결한다.

## 1. `m.category` 의 category는 어디서 나왔냐?

JPQL

```
SELECT m FROM Section06Menu m JOIN m.category c
```

여기서

```
m.category
```

는 **DB 컬럼명이 아니라 엔티티 필드명**이야.

예를 들어 Menu 엔티티가 이렇게 생겼을 거야.

```
@Entity(name = "Section06Menu")
public class Menu 
{    
	@Id    
	private int menuCode;    
	private String menuName;    
	@ManyToOne    
	@JoinColumn(name = "category_code")    
	private Category category;
}
```

여기 보면

```
private Category category;
```

있지?

JPQL은

```
m.category
```

라고 쓰면

```
Menu 엔티티 안의 category 필드
```

를 의미함.

---

SQL이면

```
join tbl_category    on tbl_menu.category_code = tbl_category.category_code
```

라고 쓰지만

JPQL은

```
JOIN m.category
```

라고 씀.

왜?

JPA는

```
테이블 중심 X객체 중심 O
```

이라서.

---

## 2. 진짜 category 필드가 없으면?

Menu 엔티티 보여줘.

```

```

만 있고

```

```

가 없다면

```

```

는 원래 에러나야 함.

## 3. N+1이 왜 발생하냐?

이걸 이해하려면 LAZY를 알아야 함.

private Category category;

라고 되어 있다고 가정.

## Native Query
- JPA에서 **네이티브 SQL** 쿼리를 **직접 실행**할 수 있도록 하는 기능
- JPA는 기본적으로 **JPQL (Java Persistence Query Language)**를 사용하여 **엔티티 기반의 쿼리를 작성**하지만, 때로는 복잡한 **SQL 쿼리**가 필요하거나, **성능 최적화**를 위해 데이터베이스에 **의존적인 쿼리**를 사용할 때가 있습니다. 
- 이럴 때 nativeQuery를 사용하면 JPQL 대신 원래의 SQL 쿼리를 사용할 수 있습니다.
- JPA에서는 손쉽게 **@Query 어노테이션**으로 커스텀 쿼리를 작성할 수 있는데, 이 때 옵션으로 **"nativeQuery = true"**로 설정한다면 **네이티브 SQL을 사용**할 수 있습니다.

# Spring Data JPA

- Spring Data Jpa는 SQL 쿼리를 직접 작성하는 대신 객체와 객체 간의 관계에 집중할 수 있게 돕는 기술, 덕분에 Java의 ORM 기술인 JPA (Java Persistence API)를 더 쉽고 편리하게 사용
- Spring Data JPA는 JPA를 더욱 편리하게 사용할 수 있도록 
	- Repository 인터페이스(CRUD 코드를 직접 안 만들어도 된다.), 
	- Query Method(메서드 이름만 지어주면 쿼리를 만들어줌), 
	- Pagination(PageRequest.of(0, 10) 하면 스프링이 알아서 limit 0,10 생성), 
	- Querydsl 연동 기능(타입 안정성(Type Safety)을 제공 등을 제공하며, 
	- EntityManager를 직접 사용하지 않고도 CRUD 기능을 구현할 수 있게 해주는 Spring의 모듈
